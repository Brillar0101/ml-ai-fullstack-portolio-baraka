# Reinforcement-Learning Attacks: Training an AI to Find the Cracks

The gradient attacks in the last story had one limitation: to compute their math, they needed to see inside the model. Against a commercial chatbot locked behind an interface, where you can only send messages and read replies, that math is unavailable. You would think that closes the door. It does not. It just means the attacker has to learn the way a person learns a locked room, by feeling around in the dark and remembering what works.

In 2024, a group of researchers led by Zhihao Lin took automated jailbreaking and gave it exactly that ability: to learn from its own failures. Their system, called PathSeeker, did not blindly try and retry the way brute-force search does. It trained a set of AI agents, using reinforcement learning, to get better at attacking with every single attempt.

Reinforcement learning is the technique behind game-playing AIs that master chess or Go by playing millions of rounds and being rewarded for good moves. PathSeeker turned that machinery on a chatbot. The agents would rephrase and mutate a harmful prompt, send it to the target, and watch the response. Then came the clever part, the reward design. They rewarded the agents partly for how rich and varied the model's reply was, and, tellingly, for any sign that the model's refusal was weakening. Every time a mutation nudged the model even slightly away from a flat no, the agent learned it was getting warmer, and pushed further in that direction.

So let's step back and name it. This is a reinforcement-learning-based attack. The defining feature is that the attacker is not a fixed procedure or a human with a clever idea. It is an agent that improves through feedback. Give it two things, the ability to send queries and a reward signal that tells it when it is getting warmer, and it will learn to jailbreak more or less on its own.

Here is the technical insight that makes this possible, and it is subtle. Every time a model refuses, it leaks information. A blunt refusal and a hesitant, hedging, almost-compliant refusal are different responses, and the difference is a gradient the attacker can climb, even though it never saw a real mathematical gradient. The reward signal, built from things like how many refusal keywords appear or how semantically close the answer is drifting toward the forbidden goal, turns that leak into a compass. A single refusal tells an attacker almost nothing. But an agent running thousands of query-and-response loops turns that trickle of signal into a detailed map of the model's boundary, and then learns the shortest path across it. The refusals themselves become the training data for defeating the refusals.

That is what separates this family from the gradient attacks. A method like GCG needs to see inside the model. A reinforcement-learning attack needs only the model's answers and a way to score them, which makes it a black-box method, usable against any commercial system you can talk to. Related versions swap in different algorithms and rewards. RL-JACK uses a custom training scheme with dense feedback from the target. A 2025 method called xJailbreak watches the model's internal representations to steer prompts toward a region that looks benign while secretly preserving the harmful intent, so the attack is stealthy as well as effective.

So how do you fix it? Because this is a cousin of the adversarial attack, it shares the deepest defense: adversarial training that hardens the refusal boundary from the inside, so there is simply less slope for the agent to climb. But reinforcement-learning attacks expose a second, more specific weakness, which is that graded refusals leak a reward signal. That points to a defense the attacker cannot easily route around: make refusals uninformative. A model that always refuses harmful requests with the same flat, consistent response gives the agent nothing to optimize, because every attempt looks equally failed, and the compass spins uselessly. On top of that, because the attack depends on thousands of probing queries, platform-level defenses matter here in a way they did not before: rate-limiting, and detecting the tell-tale pattern of an automated system hammering the same target with slight mutations, can cut the attack off before it ever converges. The fix is part robustness inside the model, and part starving the attacker of the feedback and the query volume it feeds on.

So we have gone from a human in a costume, to a hidden cipher, to a mountain of fake examples, to a math solver, to a self-improving agent. Each step removes the human from the loop and hands more of the work to the machine, and each one teaches the same defensive lesson: the more the attack automates, the more the defense has to live in the model's own robustness rather than in a filter at the door.
